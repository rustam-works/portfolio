# База знаний проекта (Project Handbook)

Этот документ объяснит «на пальцах», как устроено твоё портфолио, из чего оно состоит и за что отвечают все эти странные аббревиатуры.

---

## 1. Фундамент: Стек технологий

Твой проект — это не просто набор HTML-файлов. Это современное веб-приложение.

*   **Vite** (это «Вайт», а не «Вит»): Это твой **инструмент сборки**. Представь, что это дирижер оркестра. Он следит за тем, чтобы все твои файлы (картинки, стили, скрипты) правильно работали вместе, когда ты разрабатываешь сайт, и «упаковывает» их в максимально компактный вид, когда ты нажимаешь «Деплой».
*   **React**: Это библиотека для создания интерфейса. Вместо того чтобы писать сотни строк однообразного HTML, React позволяет собирать сайт из «компонентов» (как из конструктора LEGO).
*   **npm**: Это огромный склад бесплатных деталей (библиотек) для твоего LEGO. Когда мы пишем `npm install`, мы берем нужную деталь со склада и кладем её в проект.

---

## 2. Главные файлы (Оркестраторы)

*   **`package.json`**: Самый важный файл. Это «паспорт» твоего проекта. В нем написано:
    *   Как называется проект.
    *   Какие «детали» (зависимости) ему нужны для работы.
    *   **Scripts**: Это ярлыки для длинных команд. Например, вместо того чтобы писать сложную инструкцию по сборке, мы просто пишем `npm run deploy`, и запускается цепочка действий.
*   **`package-lock.json`**: Это «анкета» проекта для компьютера. В нем записаны точные версии всех деталей вплоть до запятой. Это нужно, чтобы и у тебя, и на сервере GitHub сайт собрался абсолютно одинаково.
*   **`vite.config.js`**: Это настройки нашего дирижера (Vite). Здесь мы подсказали ему, что у нас есть много страниц (`cv`, `tools`) и что сайт должен работать через относительные пути (чтобы ссылки не ломались).

## 3. Где живет твой код: Git и GitHub

Если проект — это книга, то **Git** — это функция «история изменений» в Google Docs.

*   **Repository (Репо)**: Папка с твоим кодом.
*   **Commit**: Это как «сохранение» в игре. Мы фиксируем состояние кода в конкретный момент времени с комментарием.
*   **Push**: Процесс отправки твоих локальных «сохранений» (коммитов) на сервер GitHub.
*   **Main**: Основная ветка с твоим «сырым» кодом (исходниками).
*   **gh-pages**: Ветка для «готовой продукции». Там лежат только те файлы, которые уже прошли обработку Vite и готовы к показу людям.

---

## 4. Деплой: Почему на GitHub это сложнее, чем на Vercel?

*   **Vercel** — это «умный» хостинг. Ты даешь ему ссылку на репо, и он сам догадывается: «Ага, это Vite, надо запустить сборку и выложить результат». Он делает всё за кулисами.
*   **GitHub Pages** — это «простой» хостинг. Он просто берет файлы и показывает их. Он не умеет сам собирать React/Vite (точнее, умеет, но это надо настраивать отдельно). 
    *   Поэтому мы используем библиотеку `gh-pages`, которая собирает сайт у тебя на компьютере в папку `dist` и сама отправляет результат в отдельную ветку.

## 5. Что мы сделали сегодня? (Кратко)

1.  **Обучили Vite**: Добавили в настройки все твои под-страницы, чтобы они попадали в сборку.
2.  **Настроили пути**: Сделали ссылки относительными (`./`), чтобы GitHub не путался в адресах.
3.  **Привязали домен**: 
    *   Создали файл **CNAME** — это визитка сайта, которая говорит серверу: «Меня зовут rustam.works».
    *   Настроили **DNS** (A-записи) — это «адресная книга» интернета, которая связывает буквы `rustam.works` с IP-адресом серверов GitHub.
4.  **Оптимизировали**: Добавили `loading="lazy"`. Это значит: «Эй, браузер, не качай картинку в самом низу страницы, пока пользователь до неё не дойдет». Это сильно экономит ресурс.

## 6. Шпаргалка на будущее

Если ты что-то поменял в дизайне или коде:
1.  **`npm run dev`** — Посмотреть изменения локально (в браузере по адресу localhost).
2.  **`npm run deploy`** — Отправить изменения в интернет. Команда сама всё соберет и обновит сайт.

**Главное правило «вайбкодинга»**: если что-то поломалось — проверь `git status` и посмотри, не забыл ли ты закомитить изменения. 

Теперь ты не просто «крутишь ручки», а понимаешь, как работает этот механизм!
